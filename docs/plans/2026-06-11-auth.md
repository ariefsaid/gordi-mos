# Plan: P1-3 — Auth (login, session, viewer profile, role surface)

- **Issue:** P1-3 (roadmap Phase 1.3). **Spec:** `docs/specs/auth.spec.md` (signed: 17 FR / 4 NFR / 14 AC).
- **ADR:** `docs/adr/0002-client-auth-architecture.md` (plain context now / TanStack at P1-4;
  schema-qualified `org_id`-free reads; admin-API e2e test users) — **read before T-005+**. Builds on
  ADR-0001 (org seam, person-first auth, fixed read posture).
- **Date:** 2026-06-11. **Branch:** `feat/p1-3-auth`.
- **Parallel design-plan:** `docs/plans/2026-06-11-auth-design.md` owns ALL visual detail
  (layout, tokens, copy placement, focus rings) for the login / recovery / orphan / home screens.
  **This plan's UI tasks (T-014..T-019) MUST NOT specify visuals** — they point `ui-implementer` at
  the design-plan and own only behavior + the failing test. If the design-plan is not yet signed when
  the UI tasks are reached, STOP and tell the Director.

## Scope guard

**IN:** `@supabase/supabase-js` client + env + minimal handwritten `shared` Database types; the
`AuthProvider` context (session, viewer resolution, `isManager` derivation, orphan detection,
sign-out); the route guard (protected redirect / login redirect / no-flash); login + recovery +
orphan + placeholder-home screens (behavior only); two pgTAP additions; four curated e2e journeys with
admin-API test-user fixtures; the Vitest coverage gate (first wired here); `config.toml` redirect
allowlist for `:5173`; the CI restructure so e2e runs against a live stack.

**OUT (do not build here):** admin/invite UI, real app shell / My Week / nav (P1-4), deep-link
return-to, profile/role editing, MFA, any directory write path. No new migrations except the
`config.toml` edit. TanStack Query is **not** added (ADR-0002 D1).

## Conventions (binding on the implementer)

- Write app code only under `mos-app/`, plus `supabase/config.toml`, `supabase/tests/10_*.sql`,
  `supabase/tests/08_claim_parsing.sql` (tag-only edit, T-012), and `.github/workflows/*.yml`.
- **TDD:** every behavior task writes the failing test FIRST (RED), then the minimum code (GREEN). The
  task's verify command must show RED before the impl sub-step and GREEN after.
- **§8 data conventions:** DB rows are snake_case; consume the DB shape directly; **never**
  `as unknown as <T>`; data-access in `src/lib/db/*` throws on error; **never send `org_id`** (RLS
  scopes it); schema-qualified reads via the `shared`-exposed PostgREST.
- AC-id tagged in the owning test's title for `grep -r AC-0NN` traceability.
- Run all `npm` commands from `mos-app/`; all `supabase` commands from repo root.
