## Architecture

A single persistent **`AppShell`** layout component wraps the four section routes via React-Router's
nested-route `<Outlet/>`, mounted under the existing `ProtectedRoute` guard. The shell owns the rail,
header, and the mobile drawer; each section route renders only its `<main>` page body. This keeps the
rail/header mounted across navigation (no re-mount flash, FR-001/005), and the per-route breadcrumb /
title / active-nav derive from `useLocation()` rather than props, so deep-links and reloads are correct
for free (FR-008/010, AC-001).

```
App
└─ RouterProvider (basename '/mos')
   ├─ RedirectIfAuthed → /login, /recovery            (unchanged)
   └─ ProtectedRoute                                   (unchanged guard; extends to new routes, FR-009)
      └─ AppShell (layout route, element)              NEW — rail + header + drawer + <Outlet/>
         ├─ index           → MyWeek                   NEW page (was Home)
         ├─ /tasks          → TasksPage                NEW empty shell
         ├─ /updates        → UpdatesPage              NEW empty shell
         ├─ /ops            → OpsPage                  NEW empty shell
         └─ *               → NotFoundPage             NEW catch-all (FR-012)
```

**Section registry as single source of truth.** One `SECTIONS` array
(`{ path, label, Icon }`) drives the rail nav, breadcrumb, document-title, and active-state — so a
nav item, its breadcrumb, its title, and its `aria-current` can never drift apart (AC-001/003/004).
The registry lives in `src/shell/sections.tsx`.

**Route-to-section resolution.** A pure `sectionForPath(pathname)` maps the current path to a section
(or `null` for unknown). The shell calls it once per render; `MyWeek`/`Tasks`/`Updates`/`Ops`/`NotFound`
each set their own `document.title` via a shared `useDocumentTitle` hook (FR-010) so a page rendered
without the shell in a test still titles correctly.

**`org_id` / RLS / migrations:** none. This issue is presentation-only over the P1-3 auth context; no
schema, no Supabase reads beyond the existing `viewer`. The one DB-adjacent change (FR-007 role
ordering) is an additive client-side `order` on an existing read — no migration, no RLS change.

**Scaling note (existing-repo lens):** `resolveViewer` already flags an O(org) full-junction read for
`isManager`; this issue does not worsen it and adds no new reads. The role-title ordering rides the
existing `person_roles` select (adds `created_at` to the column list + `.order()`), so cost is
unchanged. No new runtime dependency (NFR-002): icons are inline SVG lifted from the signed mockup.

all calling a pure `weekLabel(now: Date): WeekLabel`:
 (a) `now = new Date('2026-06-10T05:00:00Z')` (Wed 12:00 WIB) → `range: '8–14 Jun 2026'`, `today: 'Wed 10 Jun'`.

 (c) `now = new Date('2026-06-14T20:00:00Z')` — Sun 14 Jun 03:00 WIB (next day) → still

**T-005 — WIB week util impl (GREEN).** Create `src/lib/week.ts` with `weekLabel(now: Date): WeekLabel`.
Compute the WIB wall-clock date via `Intl.DateTimeFormat('en-CA', { timeZone:'Asia/Jakarta', year,
month, day, weekday })` parts (UTC+7, no DST — stable). Derive Monday-start: `dow = (jsDay + 6) % 7`
(Mon=0…Sun=6) from the WIB weekday; subtract `dow` days for the Monday, add 6 for Sunday, both as WIB
calendar dates. Format `range` as `'<d1>–<d2> <Mon> <YYYY>'` when same month/year (mockup canon
"8–14 Jun 2026"); `today` as `'<Ddd> <d> <Mon>'` (e.g. "Wed 10 Jun") using `en` short weekday/month.
Use a fixed WIB-offset arithmetic helper (add 7h to `now`, read UTC parts) so no host-tz leakage
(NFR-005). *Verify:* `npx vitest run src/lib/week.test.ts` → **passes** (AC-010).
